* **検索一覧と詳細情報を別API・別モデルとして扱う**
  検索APIだけでは名前や自己紹介などの情報が取れなかったため、一覧用と詳細用でAPIとモデルを分けることにした。

* **0件と通信エラーを分ける**
  検索結果が0件でもHTTP 200が返ってきたので、エラーではなく「該当するユーザーがいない状態」として扱う。

* **HTTPステータスを先に確認する**
  404のときは成功時とは違うJSONが返るため、先にステータスを確認してから、その結果に合ったデコードを行う。

* **nullと空文字の両方を未設定として扱う**
  未設定の項目が`null`だけでなく空文字で返る場合もあったため、どちらも未設定として扱う。

* **検索はボタンを押したときに実行する**
  未認証の検索APIは1分に10回までだったため、文字を入力するたびに検索するのではなく、検索ボタンを押したときだけ通信するようにする。

* **前の検索処理をキャンセルする**
  新しい検索をしたあとに古い検索結果が返ってくると表示がずれるため、新しい検索時に前の処理を止める。

* **検索履歴を端末に保存する**
  ローカル保存の必須要件を満たしつつ、以前検索したユーザーをもう一度探しやすくするため、検索履歴を保存する。

* **UIKit連携にSFSafariViewControllerを使う**
  GitHubのプロフィールやリポジトリをアプリ内で開く機能に使えるため、UIKit連携として採用する。
